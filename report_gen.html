1. Executive Summary & Design Philosophy

ProjectX is an enterprise-grade Zero-Trust network access platform engineered specifically to provide secure, role-based remote access to Industrial and Operational Technology (OT) devices. Traditional VPN solutions rely on granting sweeping subnet access to operators, posing massive lateral-movement risks.

By fusing the blisteringly fast cryptography of WireGuard with a robust, centrally orchestrated FastAPI backend, ProjectX completely abstracts away the complexity of managing private keys and iptables. It enforces a strictly governed Hub-and-Spoke topology where every connection is authenticated via short-lived JWTs, every packet is explicitly routed via dynamic NAT, and every user action is validated against their cryptographic identity.

2.1 Bypassing the Windows Strong Host Model (Hairpin Routing)

A significant engineering hurdle in Hub-and-Spoke VPNs on Windows Servers is the OS's native "Strong Host Model". By default, the Windows TCP/IP stack silently drops any packet that attempts to route back out the exact same interface it arrived on. This makes Peer-to-Peer pinging through a central wg0 interface impossible natively.

The ProjectX Backend API mitigates this programmatically. When generating the WireGuard tunnel configuration, the Python controller invokes native PowerShell to actively manipulate the Windows IP routing stack, enabling WeakHostSend and WeakHostReceive on the virtual adapter. This allows the packet to perform a "U-Turn" back to the destination peer. 

# Backend implementation of Hairpin enablement
subprocess.run([
    "powershell", "-Command",
    "Set-NetIPInterface -InterfaceAlias 'wg0' "
    "-Forwarding Enabled -WeakHostSend Enabled -WeakHostReceive Enabled"
], check=True)

5. Subnet Collisions and Edge NAT Translation

In industrial environments, it is incredibly common for disparate factories to use identical, overlapping internal subnets (e.g., 192.168.1.0/24). Normally, this causes catastrophic routing collisions. Because the central Hub runs on Windows Server, attempting to centrally route and NAT overlapping subnets without VRFs (Virtual Routing and Forwarding) is impossible.

ProjectX solves this implicitly via Dynamic Edge NAT. When the FastAPI backend detects a subnet collision, it assigns a unique Virtual Pool (e.g., 10.50.1.0/24) to the Factory. It automatically instructs the Central Hub to route that virtual subnet down the tunnel.

Crucially, the translation happens at the absolute Edge. The backend automatically injects native Linux iptables NETMAP scripts directly into the `PostUp` block of the Factory Gateway's WireGuard configuration file. When a Technician connects to 10.50.1.15, the packet routes to the Factory Gateway, which instantly translates the destination IP to 192.168.1.15 before forwarding it to the PLC.

6. Desktop Client Engineering & UI Synchronization

The desktop client utilizes a sophisticated C++ wrapped GUI via PyQt6. It is designed to be completely non-blocking, relying heavily on the PyQt QThread and Signal/Slot architecture.

6.1 Worker Lifecycle Management

Because network requests (like initiating a WireGuard tunnel or verifying 2FA) take time, executing them on the main GUI thread would cause the application to freeze ("Not Responding"). ProjectX offloads all I/O bound tasks to isolated QThread workers (e.g. ConnectWorker, ScanWorker).

To prevent PyQt's aggressive garbage collector from destroying running threads when a user abruptly closes the application (which historically causes the fatal QThread: Destroyed while thread is still running crash), the application utilizes a global _active_workers tracking set. During the application's closeEvent, the thread manager systematically calls terminate() and wait() on every background process, ensuring a perfectly clean exit.

6.2 Multithreaded LAN Discovery

When installed on an industrial gateway, operators must identify which PLCs and sensors are accessible. The PyQt Client features a high-performance LAN Scanner. It spawns an internal thread pool with up to 50 concurrent workers executing parallel ICMP Ping Sweeps against the local subnet. It then parses the host OS's ARP table via Regex to extract MAC addresses and maps hostnames, syncing the physical topography instantly to the cloud.

6.3 Real-Time WebSocket Synchronization & Offline Resilience

A persistent WebSocket connection binds every client application to the FastAPI server. A background WebSocketWorker runs a blocking run_forever() loop listening for JSON payloads. When an endpoint comes online, is renamed, or is deleted by an admin anywhere in the world, the backend triggers ws_manager.broadcast(). The client receives this payload and emits a PyQt Signal to the main thread, instantaneously triggering UI repaints without requiring manual API polling.

Because industrial internet connections drop frequently, the ProjectX client caches all vital configurations inside an encrypted local SQLite database. If the internet fails, the UI falls back to this localized state, preventing crashes. When connectivity is restored, queued status changes are pushed upstream seamlessly.
